# Streamlit agent skills — Roo Code bundle

Sixteen self-contained skills for building/debugging Streamlit apps with an AI
coding agent, written for a mid-size model (tested target: Qwen 3.6-27B):
each skill is ~2k tokens, load ONE per task, never all sixteen. Verified
empirically on **Streamlit 1.58** / **streamlit-aggrid 1.2** /
**streamlit-pivot 0.5** / **Altair 6.2**, and API-checked against
**Streamlit 1.55** — the
bundled `*_lab.py` files are runnable proof of every claim. The demo data is
**synthetic service-latency telemetry** (invented service/region names,
seeded random-walk latencies) — the aggrid lab's `data/` is generated by
`data/make_sample_data.py`; regenerate any time with `python make_sample_data.py`.

## The skills

| Folder | Load when… |
|---|---|
| `st-core` | ANY Streamlit work — execution model, reruns, session_state, callbacks, forms, caching, fragments. The stuck-in-rerun-loop / vanishing-state failure modes. |
| `st-layout` | Page structure — columns, containers, tabs, dialogs, sidebar, navigation, CSS targeting. |
| `st-dataframe` | Native tables — st.dataframe, column_config, sparkline columns, selection→Python, st.data_editor, Styler, Arrow dtype bugs. |
| `st-aggrid` | AG Grid grids — JsCode styling, selection fan-out, editing, grouping, the community-vs-enterprise license map. |
| `st-pivot` | BI pivots wired to Python — matrix/hierarchy/flat modes, cell-click payloads, conditional formatting. |
| `st-perspective` | Client-side exploration of huge tables — drag-drop pivots, column splits, linked viewers. CDN embed **or** the CDN-free `streamlit-perspective` route for corporate/air-gapped setups. |
| `st-altair` | Charts — the default over Plotly. mark/encode, the column:TYPE cheat sheet, MaxRowsError, click-selection→Python, layer/facet, when to switch to Plotly. |
| `st-paramtree` | Many/deep app settings — schema-driven "advanced parameters" tree (modal or inline): search, presets, JSON import/export, conditional visibility, diff-from-defaults. Ships a reusable `paramtree.py` module. |
| `st-connection` | Data access — st.connection (SQL / custom BaseConnection), cached queries with ttl, secrets.toml, and the caching rules that stop every rerun re-hitting the DB. |
| `st-testing` | Headless testing with `AppTest` — smoke-test a page, drive widgets by key, inject secrets/query_params, assert no exceptions. The self-verification skill. |
| `st-parquet` | Loading parquet/S3 data — hive partitioning (date vs aggregated), partition pruning + column projection (pushdown), S3 credentials, correct overwrites, small-files, idempotent raw→aggregate rollups. The object-store companion to `st-connection`. |
| `st-jobs` | Background work — run slow jobs (queries, batch calcs, reports) off the script thread so the app stays interactive: cache_resource'd executor, self-terminating fragment poller, the silent-failure rule for `st.*` in workers, SQLite job store, graduation path to a real queue. |
| `st-flags` | Flag icons for countries / regions / currency codes — offline Twemoji PNGs as data URIs for markdown, st.dataframe ImageColumn, and AgGrid cell renderers. Fixes the "flag emoji show as letters on Windows" trap; air-gap-safe. Ships `flag_emoji.py` + 67 assets. |
| `st-mcp-server` | Building an MCP server (FastMCP) that exposes the app's data — parquet lakes, SQL, CSV, internal APIs — as LLM-callable tools: schemas from type hints, caps/read-only guards, stdio vs streamable-http, in-memory testing — plus a DuckDB federation variant for JOINs across parquet+CSV sharing keys. Ships `mcp_data_server.py` + `mcp_duck_server.py` + tests. |
| `st-mcp-client` | Connecting an LLM to MCP servers, LangChain-first — langchain-mcp-adapters + ChatOpenAI bind_tools + a bounded tool loop (raw OpenAI-API bridge as fallback), incl. the Streamlit chat-page recipe. Ships verified tests for both paths. |
| `st-duckdb` | DuckDB inside the app — SQL (joins/aggregations) directly over parquet/CSV lakes and S3 without loading whole DataFrames: cached connection + cursor-per-op, views, httpfs/S3 secrets, register() to join widget frames with the lake, arrow/df handoff, cache_data'd queries. |

**Grid decision matrix:** flat interactive grid or selection→Python → aggrid ·
pivot/tree wired to Python → st-pivot · huge-data client-side
exploration → perspective · live range Σ/avg status bar → AG Grid enterprise
only (no free equivalent). Most "I need AG Grid" asks are covered by
`st-dataframe` for free — check there first.

## Install (Roo Code, global)

A skill is a folder named after the skill containing `SKILL.md` (the entry
point Roo reads) plus any support files. Global skills live in
`~/.roo/skills/`:

```bash
git clone https://github.com/4k7yvjv7w4-ops/streamlit-agent-skills
cd streamlit-agent-skills
mkdir -p ~/.roo/skills
cp -r st-core st-layout st-dataframe st-aggrid st-pivot st-perspective st-altair st-paramtree st-connection st-testing st-parquet st-jobs st-flags st-mcp-server st-mcp-client st-duckdb ~/.roo/skills/
```

then reload the VS Code window. (No git at work? GitHub's **Code → Download
ZIP** gives the same files.) The Settings → Skills → Add skills → **Global**
UI creates the same layout one skill at a time; the copy is equivalent and
brings the support files along. Keep the folders siblings —
`st-pivot` and `st-perspective` reference the lab in
`../st-aggrid/`. Don't copy this README into `~/.roo/skills/` (it has
no frontmatter; it's documentation, not a skill). Project-scoped alternative:
same copy into `<workspace>/.roo/skills/`.

To update later: `git pull` and re-run the `cp -r`.

## Using the skills (how they get pulled in)

Skills are **description-triggered, not manually loaded.** Roo keeps only each
skill's `name` + `description` (the frontmatter) in front of the model; when
your request matches a description, the model pulls in that skill's full
`SKILL.md` on its own. You describe the problem — the matching skill loads
itself. That's why every description is keyword-loaded with symptoms
("infinite rerun loops, values that reset on refresh, buttons that don't
work…").

**With a mid-size model (Qwen 3.6-27B), auto-selection is a weaker judgment
call — so:**

- **Lead with the symptom, not the fix.** "Values reset when I switch tabs"
  matches `st-core` better than "fix my session_state," because the
  description lists that exact symptom. Use the vocabulary in the table above
  (rerun loop · `st.dataframe` · pivot · Perspective · Altair chart).
- **Name the skill when you already know which one applies** — the reliable
  path: *"use the st-core skill — my fragment won't refresh"* or
  *"check st-aggrid for this."* This removes the guess.
- **One skill per task.** They're ~2k tokens each and built to load
  individually — don't ask it to read all sixteen. Give a focused task; the right
  one loads. The shared "choosing a grid/chart" matrix at the top of each skill
  lets the model hop onward if it landed in the wrong one.
- **Verify it loaded.** If the answer ignores the skill's specific rules (still
  suggests `use_container_width`, or `time.sleep`+`st.rerun`), it didn't pull
  the skill — re-ask and name it.

Roo's skills feature has evolved across versions, so the exact surfacing
(auto-inject vs. an explicit load step) can differ by release; the
description-driven, model-invoked model above is how these are authored to work.
Check your Roo version's Skills docs if it behaves differently.

## Streamlit 1.55 compatibility (checked 2026-07)

Every API cited by the skills was signature-checked on a real 1.55.0 install.
Everything holds on 1.55 **except three things**, flagged inline in the
skills at the point of use:

- `st.container(autoscroll=)` — 1.56+ (st-layout)
- `st.dataframe(selection_default=)` — 1.56+ (st-dataframe)
- `selection_mode="single-row-required"` — 1.56+ (st-dataframe; use
  `single-row` and treat empty selection as row 0)
- `st.fragment(parallel=)` — not on 1.55 (st-jobs mentions it only to rule it out)

The st-layout "STALE RULES" (nesting freedoms, `set_page_config` anywhere,
`width="stretch"`) all hold on 1.55; the only nesting still forbidden is
dialogs and forms — exactly what the skill says. streamlit-aggrid 1.2.x
matches the skill as written.

## First-run smoke test (do once per environment)

1. Run the labs: `python -m streamlit run ~/.roo/skills/st-core/st_core_lab.py`
   (same for st-layout, st-dataframe, st-aggrid). A broken tab = a
   claim to re-verify on your version.
2. Test the Perspective CDN (open the aggrid lab's tab 9). Blank viewer =
   proxy blocks jsdelivr → use the self-hosting workaround in that skill, or
   skip the skill.

## JupyterLab / plain-chat usage

No skill loading there — paste the ONE relevant SKILL.md into the chat/system
prompt (each fits easily in context). The table above is the routing index.
